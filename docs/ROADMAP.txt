PURSUIT OF PEACE — DEVELOPMENT ROADMAP
=======================================

Inspired by offline Discord RPG bots with addictive mechanics, designed for full JSON-based modding.
Each tier represents a complete feature set with clear deliverables and version increments.

================================================================================
TIER 1 — CORE SURVIVAL LOOP [v0.7 — COMPLETE]
================================================================================

STATUS: Foundation complete. Players can lose, die, and face consequences.

[1.1] ✓ Enemy Counter-Attack in Combat  (DONE v0.7)                          (ISSUE-001)
     
     WHAT:
     Enemies now deal damage back to the player each combat round. Previously,
     every enemy attack was ignored and combat had zero risk.
     
     HOW:
     - Each enemy in enemies.json has damage_min and damage_max fields
     - When player attacks: generate random damage between min/max
     - Deal damage to enemy, display in combat log
     - Enemy retaliates: generate random damage for enemy attack
     - Deduct from player HP, display counter-attack in log
     - If player HP <= 0, proceed to [1.2]
     
     WHY:
     Combat is now actually dangerous. Death becomes possible. This transforms combat
     from "guaranteed win" to "strategic challenge." Players must choose whether to
     fight or flee. Engagement hook: "What if I lose?"
     
     INSPIRATION FROM DISCORD BOTS:
     - Discord RPG bots use counter-attacks to create tension
     - Each turn matters because both sides take damage
     - Encourages potion use and strategic retreat
     
     IMPLEMENTATION DETAILS:
     - File: app/components/game_engine/combat_system.py
     - Method: attack() now returns both player damage AND enemy counter damage
     - Combat log shows: "You deal 7 damage! Enemy Rat strikes back for 3 HP!"
     - No AI needed; purely random rolls based on enemy stats
     
     FILES MODIFIED:
     - app/components/game_engine/combat_system.py
     - app/ui/components/complex/combat_panel.py (display counter-attacks)

[1.2] ✓ Player Death Handling  (DONE v0.7)                                   (ISSUE-002)
     
     WHAT:
     When player HP reaches 0 or below, the game ends immediately. A "Game Over"
     dialog appears showing which enemy defeated you. Player can reset state and
     return to the city without restarting the application.
     
     HOW:
     - Combat system checks: if player_hp <= 0 after enemy attack
     - Set combat_state["player_dead"] = True
     - Call GameEngine.end_combat(player_dead=True)
     - Show game-over modal: "You were defeated by [Enemy Name]!"
     - On modal dismiss: reset player to spawn location (City Gates)
     - Reset player state to last saved profile or create new with defaults
     - Return to city view so player can start new dungeon run
     
     WHY:
     Death must be consequential but not punishing. Without death, there's no
     risk. With permanent death, game feels too harsh (especially for offline).
     This balance creates tension without frustration.
     
     INSPIRATION FROM DISCORD BOTS:
     - Discord bots show "You died!" in chat, then let you retry immediately
     - No save wipes; you try again with clean slate
     - Creates "one more run" addiction loop
     
     IMPLEMENTATION DETAILS:
     - File: app/components/game_engine/combat_system.py, app/game_engine.py
     - Check: if player_current_hp <= 0 after each enemy turn
     - Modal: "Game Over\n\nDefeated by: [Enemy]\nKills this run: 3"
     - Button: "Try Again" resets state and shows city view
     - Tracking: player_state["times_died"] increments
     
     FILES MODIFIED:
     - app/components/game_engine/combat_system.py
     - app/game_engine.py
     - app/ui/components/complex/combat_panel.py (game-over modal)

[1.3] ✓ Year Rollover Trigger  (DONE v0.7)                               (ISSUE-008 / GAP-006)
     
     WHAT:
     Game time advances. An internal action counter tracks how many "actions"
     the player has taken (resting, fishing, fighting). Every 30 actions,
     one in-game year passes. At year-end, taxes come due. Unpaid taxes = exile
     (permanent game over, profile ended).
     
     HOW:
     - GameEngine tracks: action_counter (starts at 0)
     - Each significant action increments counter:
       * Defeat enemy (+1)
       * Rest at tavern (+1)
       * Fish at river (+1)
       * Take item in dungeon (+1)
       * Buy/sell item (+1)
     - Navigation actions (moving between locations) do NOT count
     - When action_counter >= 30 (ACTIONS_PER_YEAR):
       * Call GameEngine.process_year_rollover()
       * Increment player_state["year"]
       * Check: if player_state["tax_paid"] == False:
         - Show exile screen: "You could not pay your taxes! You are exiled forever!"
         - Profile marked as dead; cannot be resumed
         - Reload latest save or start new profile
       * Reset tax_paid to False for new year
       * Reset action_counter to 0
     
     WHY:
     Time pressure is the ultimate engagement hook. Without year/time, taxes are
     cosmetic. With them, players feel urgency: "I need X gold before year-end."
     This drives repeated play sessions and creates natural stopping points.
     
     INSPIRATION FROM DISCORD BOTS:
     - Discord bots use "daily resets" for urgency (e.g., "Daily quest resets in 3h")
     - Time pressure makes casual decisions matter
     - "Gotta get it done before deadline" is a proven addiction mechanic
     
     IMPLEMENTATION DETAILS:
     - File: app/game_engine.py, app/components/game_engine/state_manager.py
     - Configuration: data/dungeon/config.json includes "ACTIONS_PER_YEAR": 30
     - Rollover check happens after EVERY action
     - Tax amount: data/economy/prices.json["tax"]
     - Player must visit City Hall and use "pay_taxes" action before year-end
     - If still unpaid at rollover: exile screen, game-over dialog
     - Tracking: player_state["years_survived"] (for stats)
     
     FILES MODIFIED:
     - app/game_engine.py (action counter and rollover check)
     - app/components/game_engine/location_actions.py (pay_taxes action)
     - app/components/game_engine/view_builder.py (display year/tax status)
     - data/dungeon/config.json (ACTIONS_PER_YEAR setting)

TIER 2 — CONSUMABLE & EFFECT SYSTEM [v0.8]
================================================================================

STATUS: Items become usable. Inventory gains agency.

[2.1] Item Use / Consumable Effects System
     
     WHAT:
     Players can now "Use" consumable items from their inventory. Each item has
     an effect_id that maps to an effect definition. Using an item applies its
     effect (heal HP, remove debuff, gain temporary buff) and removes the item
     from inventory.
     
     HOW:
     
     STEP 1: Define effects in data/consumables/effects.json:
       {
         "heal_5_hp": {
           "type": "heal",
           "target_stat": "hp",
           "value": 5,
           "is_stacking": false
         },
         "remove_poison": {
           "type": "remove_buff",
           "buff_id": "poisoned"
         },
         "gain_strength": {
           "type": "add_buff",
           "buff_id": "strength_3",
           "duration": "one_run",
           "stacking": false
         }
       }
     
     STEP 2: Add effect_id to items in data/dungeon/items/items.json:
       {
         "id": "item_bandage_01",
         "name": "Cloth Bandage",
         "type": "consumable",
         "effect_id": "heal_5_hp",
         "effect_text": "Restores 5 HP",
         "value": 6
       }
     
     STEP 3: UI Implementation:
       - Inventory panel shows list of items
       - Consumables have "Use" button (weapons/armor show "Equip" instead)
       - On click "Use": apply effect, show "Used Bandage! HP +5", remove item
       - In combat: can use consumable before/after attack turn
     
     STEP 4: Effect Resolution:
       - EffectResolver.apply(effect_def, player_state) -> new_player_state
       - Healing: player_hp += value (capped at max_hp)
       - Buff application: add to player_state["buffs"] array
       - Remove buff: filter out matching buff_id from buffs array
       - Log action: "You used Bandage!"
     
     WHY:
     Consumables create strategic depth. Do you use potions now or save them?
     Makes inventory meaningful instead of display-only. Also creates economy:
     players craft/buy potions, use them, need more. Engagement hook: "Use potion
     before boss fight."
     
     INSPIRATION FROM DISCORD BOTS:
     - Discord bots have `/use potion` command that heals you
     - Limited potions create tension: "Do I have enough for the boss?"
     - Item usage is a core feedback loop
     
     IMPLEMENTATION DETAILS:
     - File: app/components/game_engine/effect_resolver.py (NEW FILE)
     - File: app/components/game_engine/location_actions.py (NEW: use_item method)
     - Effects are pure data, no hardcoding in Python
     - Healing effects: HP capped at max_hp (no overflow)
     - Buff effects: duration stored in buff object
     - Stacking: flag controls if multiple of same buff can exist
     - In combat: use_item_in_combat action available in combat panel
     
     DESIGN CONSIDERATIONS:
     - Consumables in combat are powerful but limited (makes them valuable)
     - Can only use items you have (inventory check)
     - Effect text shown in UI: "[Item Name] was used! [Effect Text]"
     - No "undo": once consumed, it's gone (teaches consequence)
     
     FILES MODIFIED:
     - app/components/game_engine/effect_resolver.py (NEW)
     - app/components/game_engine/location_actions.py
     - app/ui/components/complex/inventory_panel.py (add Use button)
     - data/consumables/effects.json (NEW)
     - data/dungeon/items/items.json (add effect_id field)

[2.2] Buff / Debuff Application System
     
     WHAT:
     Buffs and debuffs now actually apply their effects to player stats. Previously,
     buffs were stored but ignored. Now:
     - Strength buff: +3 damage to all attacks
     - Poison debuff: -1 HP per combat round
     - Cleanliness buff: +5 max HP while active
     - Curse debuff: -50% gold from enemies while active
     
     HOW:
     
     STEP 1: Buff structure in player_state:
       player_state["buffs"] = [
         {
           "id": "strength_3",
           "source": "potion_strength",
           "duration": "one_run",
           "duration_remaining": 1,
           "stat_mods": {"damage": 3, "max_hp": 0},
           "label": "Strength"
         }
       ]
     
     STEP 2: At combat start, apply all active buffs:
       - Loop through player_state["buffs"]
       - For each buff: apply its stat_mods to derived stats
       - Example: if buff has {"damage": 3}, player_damage_roll += 3
     
     STEP 3: At combat end or rest:
       - Loop through buffs and decrement duration_remaining
       - Remove buffs with duration_remaining <= 0
       - Show: "Your Strength buff has worn off."
     
     STEP 4: Debuffs reduce stats (negative mods):
       - Poison debuff: {"hp": -1} applies per turn
       - Curse debuff: {"gold_earned": -0.5} (multiplier)
       - Cursed item debuff: {"max_hp": -10}
     
     WHY:
     Status effects create tactical depth. Do I use the potion now to get buff,
     or save it for later? Debuffs make certain enemies dangerous (poison enemies
     make you want antidotes). Buffs from baths/potions become worth seeking.
     
     INSPIRATION FROM DISCORD BOTS:
     - Discord bots have `/status` showing active effects
     - Buffs encourage replaying content ("one more run for strength buff")
     - Debuffs make enemies unique and require counter-strategies
     
     IMPLEMENTATION DETAILS:
     - File: app/components/game_engine/buff_system.py (NEW FILE)
     - Method: BuffSystem.apply_buffs(player_state, combat_state) -> modified_combat_state
     - Method: BuffSystem.tick_buffs(player_state) -> new_player_state (decrement durations)
     - Method: BuffSystem.add_buff(player_state, buff_def, source) -> new_player_state
     - Method: BuffSystem.remove_buff(player_state, buff_id) -> new_player_state
     - Called from: CombatSystem.start_combat(), LocationActions.rest(), etc.
     - Buff UI display: show active buffs in player panel (top right)
     - Buff persistence: saved in profile so buffs survive between sessions
     - Buff expiration: at rest or dungeon exit, tick durations
     
     DESIGN CONSIDERATIONS:
     - Debuffs can be removed by antidote consumables
     - Some buffs are permanent (equipment-based, see Tier 3)
     - Some buffs are one-run (session-based, removed at dungeon exit)
     - Stacking: by default, only one of each buff type active
     - UI indicator: [+] for positive buffs (green), [-] for negative (red)
     
     FILES MODIFIED:
     - app/components/game_engine/buff_system.py (NEW)
     - app/components/game_engine/combat_system.py (apply buffs on combat start)
     - app/components/game_engine/location_actions.py (tick buffs on rest)
     - app/components/game_engine/state_manager.py (persist buffs in save)
     - app/ui/components/complex/player_panel.py (display buffs)
     - data/consumables/effects.json (buff effects already defined in [2.1])

     STEP 4: Equip/unequip logic:
       def equip_item(player_state, item_id, slot_type):
         item = find_item(item_id, items.json)
         if slot_type == "weapon" and item.type != "weapon":
           return player_state, "Can only equip weapons here."
         # Remove old equipped item if exists
         if player_state[f"equipped_{slot_type}"]:
           old_item_id = player_state[f"equipped_{slot_type}"]["item_id"]
           player_state["inventory"].append(old_item_id)
         # Equip new item
         player_state[f"equipped_{slot_type}"] = {
           "item_id": item_id,
           "current_durability": item.durability,
           "max_durability": item.durability
         }
         player_state["inventory"].remove(item_id)
         return player_state, f"Equipped {item.name}!"
     
     WHY:
     Equipment progression is the core of character growth. Having gear creates
     replayability: "I want to find better equipment." Equipment makes exploration
     purposeful and gives dungeon loot value beyond just gold.

[3.2] Combat Damage Scaling with Equipment
     
     WHAT:
     When calculating combat damage, apply equipped weapon's stat_bonus.
     When reducing incoming damage, apply equipped armor's stat_bonus as reduction.
     This makes equipment directly visible in combat mechanics.
     
     HOW:
     
     STEP 1: Calculate player attack damage:
       base_player_damage = 1 + random(0, 5)  // baseline
       if equipped_weapon:
         weapon_damage_bonus = equipped_weapon["stat_bonus"]["damage"]
         total_damage = base_player_damage + weapon_damage_bonus
       else:
         total_damage = base_player_damage
     
     STEP 2: Calculate enemy attack damage received:
       base_enemy_damage = enemy.damage_min + random(0, enemy.damage_max - enemy.damage_min)
       if equipped_armor:
         armor_defense = equipped_armor["stat_bonus"]["defense"]
         damage_received = max(1, base_enemy_damage - armor_defense)
       else:
         damage_received = base_enemy_damage
     
     STEP 3: Apply damage:
       player_hp -= damage_received
       display: "Enemy deals 6 damage. Your armor absorbs 3! You take 3 HP."
     
     WHY:
     Makes equipment meaningful in real-time combat. Players see their gear
     working. Incentivizes equipment collection and upgrades. Creates progression
     feeling: "I hit for 12 now instead of 5!"
     
     INSPIRATION FROM DISCORD BOTS:
     - Discord bots show "/attack" with your total damage (including gear)
     - Damage changes feel impactful and encourage gear upgrades
     - Equipment progression is visible and satisfying
     
     IMPLEMENTATION DETAILS:
     - File: app/components/game_engine/combat_system.py
     - Method: calculate_player_damage(combat_state, equipped_weapon) -> int
     - Method: calculate_incoming_damage(enemy_damage, equipped_armor) -> int
     - Combat log shows breakdown: "You deal 8 damage (3 base + 5 sword)!"
     - Armor shows: "Enemy deals 7 damage (10 base - 3 armor absorbed) = 7 taken"
     - Armor can't reduce damage below 1 (no invulnerability)
     
     DESIGN CONSIDERATIONS:
     - Damage scaling is linear (sword +5 = 5 extra damage always)
     - No capping: can combine with buff system for high damage potential
     - Armor is defensive only (doesn't boost attack)
     - Different weapons have different bonuses (swords +5, axes +7, etc)
     - Can fight unequipped (just lower damage)
     
     FILES MODIFIED:
     - app/components/game_engine/combat_system.py (damage calculation)
     - app/components/game_engine/view_builder.py (display damage breakdown)

[3.3] Equipment Durability System
     
     WHAT:
     Equipped items have durability. Each combat attack decrements durability.
     When durability reaches 0, item breaks and unequips automatically. Breaking
     items creates a maintenance cost (must repair or find new gear). This drives
     the economy loop: earn gold -> repair equipment -> get stronger.
     
     HOW:
     
     STEP 1: Equipment has durability:
       equipped_weapon = {
         "item_id": "iron_sword_01",
         "current_durability": 42,
         "max_durability": 50
       }
     
     STEP 2: On attack, decrement durability:
       def attack(self, combat_state):
         ...
         player_damage = calculate_player_damage(...)
         enemy.hp -= player_damage
         # Decrement equipped weapon durability
         if self.equipped_weapon:
           self.equipped_weapon["current_durability"] -= 1
           if self.equipped_weapon["current_durability"] <= 0:
             show_message(f"Your {weapon_name} broke!")
             player_state["equipped_weapon"] = None
         ...
     
     STEP 3: Durability display:
       - UI shows durability bar: [=========>  ] 42/50
       - Low durability warning (< 20%): "Your sword is badly damaged! (2/50)"
       - Color feedback: green -> yellow -> red as durability drops
     
     WHY:
     Durability creates a repeating gold sink. Without maintenance costs, once
     you get good gear, economy loops stop. Durability forces players to keep
     engaging with shops and repairs. Psychological effect: "My gear is dying,
     I need gold!" drives farming.
     
     INSPIRATION FROM DISCORD BOTS:
     - Some Discord RPG bots have durability (rare, but effective)
     - Durability creates urgency without being unfair
     - Maintenance loop keeps players engaged
     
     IMPLEMENTATION DETAILS:
     - File: app/components/game_engine/combat_system.py (decrement on attack)
     - File: app/ui/components/complex/inventory_panel.py (durability display)
     - Durability only decrements on player attack (not on defend/flee)
     - Only equipped items lose durability (inventory items safe)
     - Different gear types degrade at same rate (1 per attack)
     - Breaking shows message and automatically unequips
     - Player can still fight with broken item (just at -50% damage?) or must repair
     
     DESIGN CONSIDERATIONS:
     - Durability loss is deterministic (always 1 per attack)
     - Breaking at 0 prevents negative durability
     - Armor also degrades (at same rate as weapon)
     - Degradation is visible to keep player aware
     - Creates natural stopping point for dungeon runs ("better leave before breaking")
     
     FILES MODIFIED:
     - app/components/game_engine/combat_system.py
     - app/ui/components/complex/inventory_panel.py (durability bar)

       class EventManager:
         def __init__(self, events_data):
           self.events = events_data
         
         def get_location_events(self, location_id):
           return [e for e in self.events if e["location_id"] == location_id]
         
         def trigger_location_events(self, player_state, location_id):
           events = self.get_location_events(location_id)
           for event in events:
             if random() < event["chance"]:
               return self.resolve_event(player_state, event)
           return player_state, None
         
         def resolve_event(self, player_state, event):
           outcome = random.choice(event["outcomes"])
           player_state = apply_effects(player_state, outcome["effects"])
           return player_state, outcome["text"]

       - On showing location view: display event text if triggered
       - Events are optional display layer (don't break core gameplay)

        def check_fishing(player_state):
          if not player_state["fishing_active"]:
            return
          time_since_cast = current_time() - player_state["fishing_start_time"]
          if time_since_cast % 10 == 0:  // every 10 seconds
            if random() < 0.6:  // 60% catch rate
              fish = random_choice(FISH_POOL)
              player_state["inventory"].append(fish)
              player_state["fish_caught_this_session"] += 1
              notify_player(f"You caught a {fish['name']}!")

          if not player_state["fishing_active"]:
            return player_state, "You're not fishing."
          player_state["fishing_active"] = False
          total = player_state["fish_caught_this_session"]
          return player_state, f"You reel in and caught {total} fish!"
      
      WHY:
      Idle activities are addictive engagement hooks. Players can "leave game running"
      and accumulate resources AFK. Creates FOMO: "I should go check fishing results!"

      INSPIRATION:

          recipe = find_recipe(recipe_id)
          for ingredient in recipe["ingredients"]:
            if ingredient["item_id"] not in player_state["inventory"] * ingredient["quantity"]:
              return player_state, "Missing ingredients."
          # Remove ingredients
          for ingredient in recipe["ingredients"]:
            for _ in range(ingredient["quantity"]):
              player_state["inventory"].remove(ingredient["item_id"])
          # Add result
          for _ in range(recipe["result_quantity"]):
            player_state["inventory"].append(recipe["result_item"])
          return player_state, f"Crafted {recipe['result_item']}!"

      seed = different dungeon. Creates infinite dungeon replayability.
      
      HOW:

      - Player can share dungeon seeds with community
      - Each dungeon run: new random seed (unless player selects specific)
      
      WHY:
      Procedural generation eliminates replay fatigue. Same dungeon becomes
      boring, but randomized dungeons feel fresh. Infinite content without
      manual level design.
